    Architecture: daemon, IPC, and connection lifecycle

    This document explains the overall design of VPNConnect: how the daemon, CLI, and UI relate to each other, how they communicate, and how a connection moves through its lifecycle.

    The daemon/server model

    VPNConnect uses a privileged daemon pattern. A long-running server process owns the VPN interface with root privileges, while lightweight client processes (CLI commands, the UI) communicate with it over a local TCP socket.

     ┌──────────────────────┐     JSON over TCP      ┌──────────────────────────┐
 │  vpnconnect connect  │ ──────────────────────► │  vpnconnect server       │
 │  vpnconnect status   │ ◄────────────────────── │  (daemon, runs as root)  │
 │  vpnconnect ui       │                         │  owns TUN/TAP interface  │
 └──────────────────────┘                         └──────────────────────────┘
        clients                                         server / daemon

    Why this split?

    Creating and managing TUN/TAP interfaces requires elevated privileges on all supported platforms. The daemon runs once as root (or SYSTEM on Windows) and stays running; unprivileged users interact with the VPN without ever requiring sudo themselves.

    Server info file

    When the daemon starts it binds to a random local TCP port and writes a small JSON file that clients read to locate it:

    Platform Path
    Linux /run/vpnconnect/server.json
    macOS /var/run/vpnconnect/server.json
    Windows %ProgramData%\vpnconnect\server.json

    The file contains the port number and a random auth token generated at startup. Every client request must include this token in the message headers.

    IPC protocol

    Communication is a line-delimited JSON request/response protocol sent over the local TCP connection.

    Request message

    {
  "command": "connect",
  "headers": { "token": "<auth-token>" },
  "body": { "params": "<serialized-session-json>" }
}

    Response message

    {
  "status": "ok",
  "headers": {},
  "body": { "result": "connected" }
}

    A status of "error" means the body["error"] field contains an error string.

    Supported commands: connect, disconnect, status, statistics.

    Package map

    internal/
├── pulse/           Pulse protocol implementation
│   ├── auth/        IF-T/TLS + EAP authentication
│   ├── esp/         ESP/DTLS data tunnel, key derivation
│   ├── tls/         TLS-only data channel handler
│   ├── config/      Network config parsing (AVP attributes)
│   ├── avp/         Attribute-Value Pair encoding/decoding
│   ├── ift/         IF-T packet framing
│   ├── eap/         EAP message types
│   └── pac/         PAC file extraction for DNS domains
├── tuntap/          TUN/TAP interface management (platform abstraction)
├── platformnet/     OS-specific network operations (routes, DNS, interfaces)
├── mfa/             Browser automation for Microsoft SSO/Azure AD
├── profile/         VPN profile storage with OS keychain integration
├── server/          Daemon process (TCP server, command dispatcher)
├── session/         Connection parameter serialisation for IPC
├── protocol/        JSON message framing shared by server and clients
├── icons/           Embedded application icons
└── log/             Structured logger

    Connection lifecycle

    A full connection passes through these stages in order. The order is critical: certain steps must happen before others to avoid routing loops and kernel ordering constraints.

    connect()
│
├─ 1. Authenticate
│      IF-T/TLS handshake → EAP flow → receive session token / DSID
│
├─ 2. Receive configuration
│      Parse AVP attributes: IP, netmask, MTU, DNS, routes, PAC
│
├─ 3. Determine ESP server IP  (if ESP is enabled)
│
├─ 4. ESP handshake  ◄── must happen BEFORE TUN creation
│      Negotiate keys, set up UDP ESP tunnel
│
├─ 5. Add route exception for VPN server IP
│      Pins the server to the default gateway so traffic to it
│      is NOT routed back into the VPN tunnel
│
├─ 6. Create TUN/TAP interface
│
├─ 7. Configure interface
│      Assign IP/mask, set MTU, add routes, configure DNS
│
└─ 8. Start data channel handler
       ESP handler (UDP) or TLS handler (TCP fallback)

    Disconnection reverses these steps: the data channel is stopped, routes are removed, the TUN/TAP interface is torn down, and a logout packet is sent to the server (see Logout and session termination).

    Platform abstraction

    Platform-specific code lives in separate files selected at build time using Go build tags:

    Area Linux macOS Windows
    TUN/TAP tuntap/platform_linux.go tuntap/platform_darwin.go tuntap/platform_windows.go
    Routes/DNS/interfaces platformnet/platformnet_linux.go platformnet/platformnet_darwin.go platformnet/platformnet_windows.go
    Interface naming tun0, tun1, … utun0, utun1, … wintun
    DNS configuration systemd-resolved DNSZones winipcfg (wireguard/windows)
    Wintun driver — — wintun.dll

    The tuntap.Manager interface and the platformnet package provide a unified API that the rest of the codebase uses; platform differences are hidden behind those abstractions.

    Standalone mode

    When vpnconnect connect --standalone is used, no daemon is involved. The CLI process itself creates the TUN interface and manages the data channel, blocking until interrupted. This mode:

    • Requires the process to have root/administrator privileges directly.
    • Is useful on systems where installing the daemon is not practical.
    • Does not support status or statistics queries from a separate terminal (because there is no daemon to query).

    See How to connect in standalone mode for usage details.
