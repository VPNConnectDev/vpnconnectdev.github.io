    Pulse protocol internals

    This document explains how VPNConnect implements the Juniper Pulse Secure VPN protocol.

    Overview

    The internal/pulse package provides a client-side implementation of the Pulse Secure VPN protocol. It covers:

    • IF-T/TLS authentication (username/password or DSID cookie)
    • EAP authentication (Identity, GTC, Juniper expanded types)
    • Network configuration parsing and application (IPv4, DNS, MTU, routes, PAC)
    • TUN/TAP interface creation and management
    • TLS-only data channel (fully implemented and production-ready)
    • ESP/DTLS data channel (implemented; not fully validated in all environments)
    • Platform-specific routing for endpoint exceptions (Linux, macOS, Windows)
    • Live connection status and statistics

    Feature completeness

    Feature Status
    TLS-only data channel Production-ready
    ESP/DTLS data channel Implemented; not guaranteed in all environments
    IPv4 Fully supported
    IPv6 Not supported
    Host checker / posture Not supported
    Advanced Pulse features Not supported

    Authentication

    Authentication uses the IF-T/TLS protocol (IF-T over TLS). The auth.Authenticator handles the full handshake:

    1. Open a TLS connection to the VPN endpoint.
    2. Send an HTTP Upgrade request to switch the connection to IF-T/TLS.
    3. Negotiate the IF-T protocol version.
    4. Send client information (hostname, local IP address).
    5. Perform EAP authentication:
    6. Identity: client announces its identity.
    7. GTC: generic token / password challenge.
    8. Juniper expanded: Juniper-specific EAP sub-types for session and cookie handling.
    9. Handle DSID cookie shortcuts: if the server sends a configuration packet early (because a valid DSID cookie was presented), the EAP flow is skipped.

    Two authentication modes are supported:

    • Username/password: credentials are read from the profile (or OS keychain) and sent in the EAP-GTC response.
    • DSID cookie: a pre-obtained session cookie (e.g. from the MFA flow) is presented directly. This is the faster path for MFA-enabled profiles.

    Network configuration

    After authentication succeeds the server sends a series of AVP (Attribute-Value Pair) packets containing the client's network configuration. The connection phase ends when a packet of type 0x8f is received.

    Parsed attributes include:

    Attribute Description
    IPv4 address and netmask Assigned address for the TUN interface
    MTU Maximum Transmission Unit for the tunnel
    DNS servers Primary and secondary resolver IPs
    DNS search domains Suffixes appended to unqualified queries
    Include routes Subnets to route through the VPN (split tunnelling)
    Exclude routes Subnets to bypass the VPN
    PAC file / PAC URL Proxy auto-config for extracting internal domains

    For the full list of attributes and their support status, see Pulse protocol attributes.

    PAC file processing extracts internal domain names from the PAC JavaScript and adds them as DNS search domains. See PAC file parsing for details.

    Data channels

    VPNConnect supports two data channel modes:

    TLS-only mode

    Packets are encapsulated inside the existing IF-T/TLS connection. The tls.Handler reads IP packets from the TUN interface and writes them as IF-T frames to the TLS socket, and vice versa.

    This mode is always available and is used as a fallback when ESP is unavailable or not negotiated.

    ESP / DTLS mode

    A parallel UDP tunnel is established using ESP (Encapsulating Security Payload). The esp package handles key derivation, packet encryption (AES-256 by default), integrity protection (SHA-1 HMAC), and anti-replay sequence numbering.

    ESP offers better throughput and lower latency than TLS because it avoids TCP head-of-line blocking. However, it requires UDP connectivity to the server on port 4500 (or the port specified by attribute 0x4016).

    If the ESP handshake fails or times out (attribute 0x4017 controls the timeout), vpnconnect falls back to TLS-only mode automatically.

    Routing design

    Two host routes are added to the system routing table before the TUN interface is brought up:

    1. VPN server IP → default gateway: ensures that control-plane traffic (TLS connection) is not routed into the VPN tunnel, which would cause a routing loop.
    2. ESP server IP → default gateway (if different from the TLS server): same reason, for the UDP data channel.

    These routes are removed on disconnect.

    All other traffic routing is determined by the include/exclude routes from the server configuration. VPNConnect applies these as direct ip route (Linux), route add (macOS), or RouteData (Windows) entries on the TUN/TAP interface.

    Key types

    Type Description
    Protocol Interface for VPN protocol implementations
    Client Main struct; implements Protocol
    ConnectionConfig All connection parameters and per-connection options
    ConnectionStatus Reports current state (connecting, connected, disconnected, error)
    auth.Authenticator Handles IF-T/TLS and EAP authentication
    tls.Handler Forwards packets between TUN and TLS socket
    esp.Tunnel Manages the ESP UDP data channel

    Limitations

    • IPv6 only for routing: IPv6 is supported for split-include/exclude routes and DNS, but the TUN interface itself is IPv4-only. Cross-protocol tunnelling (IPv6 inside an IPv4 ESP tunnel) depends on "Unstupid mode" support from the server (0x4024).
    • No host checker: Pulse Secure's host-checker / posture-assessment feature is not implemented.
    • ESP not production-validated: The ESP path is functionally complete but has not been tested in all real-world server configurations.
