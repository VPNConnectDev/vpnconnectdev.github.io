    PAC file parsing

    This document explains how VPNConnect processes Proxy Auto-Config (PAC) files received from Pulse Secure VPN servers and why this is necessary for correct DNS and split-tunnel behaviour.

    What is a PAC file in this context?

    Pulse Secure VPN servers sometimes deliver a PAC (Proxy Auto-Config) file to clients as part of the network configuration exchange. A PAC file contains a JavaScript function — FindProxyForURL — that, for each destination, returns either "PROXY <address>" (route via a proxy) or "DIRECT" (connect directly).

    For enterprise networks the PAC file encodes the topology of the internal network: which domains and addresses are internal (and therefore reachable only through the VPN) and which are public (and should go directly to the internet).

    VPNConnect does not run the JavaScript. Instead, it statically parses the PAC source to extract domain patterns, which are then used to configure DNS search domains on the VPN interface.

    Why extract DNS search domains from PAC?

    Some Pulse Secure deployments do not send explicit DNS search domains (AVP attribute 0x4006). Their PAC file is the only place where the list of internal domains is expressed. If vpnconnect did not parse the PAC file, DNS queries for internal resources would not be routed through the VPN's DNS servers.

    By extracting PROXY target domains from the PAC file and adding them as DNS search domains, vpnconnect ensures that names like internal.example.com resolve via the VPN DNS servers rather than the upstream public resolver.

    How the parser works

    The PACParser in internal/pulse/pac uses regular expressions to find conditional blocks in the PAC JavaScript and identify their associated actions.

    Extraction steps:

    1. Find if/else if blocks containing return "PROXY ..." or return "DIRECT".
    2. Inside those blocks, look for calls to domain-matching functions:
    3. dnsDomainIs(host, ".example.com")
    4. shExpMatch(host, "*.internal.corp")
    5. Extract the domain patterns from the function arguments.
    6. Classify each pattern as a proxy route (internal) or direct route (public).

    Domain compression

    Raw PAC files from enterprise deployments often contain hundreds of entries that are redundant when expressed as DNS search domains. The parser applies a compression step to reduce noise:

    1. Group by suffix: count how many distinct sub-domain patterns share a common suffix (e.g. a.example.com, b.example.com, c.example.com all share .example.com).
    2. Threshold promotion: if the count of sub-domains for a suffix exceeds a threshold, replace all of them with the root suffix (example.com).
    3. Remove dominated entries: once a broader pattern is present (e.g. example.com), remove any specific entries that are already covered (a.example.com is covered by example.com).

    The result is a minimal, non-redundant set of domain suffixes that accurately represents the internal network scope.

    Integration with DNS configuration

    When the Pulse client receives a PAC payload (attribute 0x4009) or a PAC URL (attribute 0x4023):

    1. A PACParser is instantiated with the PAC content.
    2. TransformPAC() is called to run the extraction and compression.
    3. GetDirectRoutes() returns the internal domain list (the DIRECT routes in the PAC, which correspond to publicly accessible names; the PROXY routes correspond to the VPN-internal names).
    4. The resulting domains are prepended to the DNS search domain list configured on the TUN interface.

    Platform-specific DNS configuration (systemd-resolved on Linux, DNSZones on macOS, winipcfg on Windows) then ensures these domains are resolved via the VPN DNS servers.

    Limitations

    • The parser only handles static domain patterns. Dynamic JavaScript logic (function calls, variable interpolation) is not evaluated.
    • Sufficient for most enterprise Pulse Secure deployments, which use straightforward domain lists.
    • The extraction is best-effort: unusual PAC patterns that deviate from the common dnsDomainIs / shExpMatch idioms may not be detected.
