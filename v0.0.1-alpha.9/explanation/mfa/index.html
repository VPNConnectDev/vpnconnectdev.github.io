    MFA browser automation

    This document explains how VPNConnect handles Multi-Factor Authentication (MFA) for enterprise VPN endpoints that use Microsoft Azure AD / Office 365 sign-in.

    Why browser automation?

    Pulse Secure VPN endpoints integrated with Microsoft SSO do not expose a machine-readable API for the authentication step. The only way to authenticate is through the browser-based sign-in flow. VPNConnect opens a real Chrome browser (via the chromedp library) and, optionally, automates the credential-entry steps so the user only needs to approve a push notification, enter a TOTP code, or touch a security key.

    The browser opens visibly so the user can complete the MFA prompt (authenticator app push, SMS code, etc.). Once the authentication succeeds, vpnconnect extracts the DSID session cookie from the browser and closes the window.

    Automation modes

    VPNConnect supports three modes for the browser-based authentication flow, controlled by --mfa-automation-mode.

    Manual mode (default — omit --mfa-automation-mode)

    The browser navigates to the VPN endpoint URL and waits. The user fills in all credentials and completes any MFA challenge themselves. This works with any identity provider — Microsoft SSO, other SAML/OIDC providers, or custom login pages.

    SSO mode (--mfa-automation-mode sso)

    The browser navigates to the VPN endpoint and automatically fills in the Microsoft SSO username and password fields. The user only needs to complete the MFA challenge (push notification, TOTP code, etc.). This mode is tailored to the Microsoft Azure AD / Office 365 login page selectors and may break if Microsoft changes the page layout.

    Enable it permanently in the profile:

    vpnconnect profile add --name work --endpoint https://vpn.example.com \
  --username alice@example.com --mfa --mfa-automation-mode sso

    Or override for a single invocation:

    vpnconnect connect --profile work --mfa-automation-mode sso
vpnconnect mfa --profile work --mfa-automation-mode sso

    FIDO2 mode (--mfa-automation-mode fido2)

    The browser navigates to the VPN endpoint and automatically clicks "Sign-in options" on the Microsoft Entra login page, then selects the passkey / security key tile. The user handles the native WebAuthn dialog (PIN entry + key touch) manually. This mode is tailored to Microsoft Entra DOM selectors and may break if Microsoft changes the page layout.

    Enable it permanently in the profile:

    vpnconnect profile add --name work --endpoint https://vpn.example.com \
  --username alice@example.com --mfa --mfa-automation-mode fido2

    Or override for a single invocation:

    vpnconnect connect --profile work --mfa-automation-mode fido2
vpnconnect mfa --profile work --mfa-automation-mode fido2

    The DSID cookie

    The DSID cookie is the Pulse Secure session token. Once obtained, it is passed directly to the VPN authentication step in place of a username/password, bypassing the MFA challenge for the duration of the session.

    When the connect command detects that a profile has MFAEnabled: true and no cached DSID is available, the MFA flow is triggered automatically before the connection request is sent to the daemon.

    The mfa command can be used to pre-obtain the DSID cookie without immediately connecting:

    vpnconnect mfa --profile work

    Authentication flow

    1.  Open Chrome (non-headless, so user sees the window)
2.  Navigate to the VPN endpoint URL
3.  Handle pre-auth proceed button (if present)          →  #sn-preauth-proceed_2
4.  [sso only] Fill in username                          →  //input[@name="loginfmt"]
5.  [sso only] Click submit                              →  //input[@type="submit"]
6.  [sso only] Fill in password                          →  #passwordInput[name="Password"] or input[name="passwd"]
7.  [sso only] Click Microsoft sign-in button            →  *[id*=idSIButton9]
8.  [fido2 only] Click "Sign-in options"                 →  div[data-test-id="signinOptions"]
9.  [fido2 only] Select passkey/security key tile        →  div[data-test-cred-id="7"]
10. User completes MFA (push / TOTP / WebAuthn key touch)
11. Handle post-auth proceed button (if present)         →  #sn-postauth-proceed_2
12. Handle session-limit dialog (if present)             →  #postfixSID_1, #btnContinue
13. Extract DSID cookie
14. Close the browser and return DSID to caller

    Steps 3, 11, and 12 are conditional — not all deployments show these UI elements. Steps 4–7 only run in sso mode; steps 8–9 only run in fido2 mode. In manual mode the user fills in all credentials themselves.

    Selectors reference

    Element Selector Mode
    Pre-auth proceed button #sn-preauth-proceed_2 all (optional)
    Username field //input[@name="loginfmt"] sso
    Submit button //input[@type="submit"] sso
    Password field #passwordInput[name="Password"] sso (variant 1)
    Password field (alt) input[name="passwd"] sso (variant 2)
    Sign-in button *[id*=idSIButton9] sso
    Sign-in options button div[data-test-id="signinOptions"] fido2
    Sign-in options button (alt) div[role="button"][aria-label="Sign-in options"] fido2
    Passkey / security key tile div[data-test-cred-id="7"] fido2
    Post-auth proceed button #sn-postauth-proceed_2 all (optional)
    Session-limit radio #postfixSID_1 Shown when max sessions reached
    Continue button #btnContinue Session-limit confirmation

    Timeout

    The MFA handler waits up to 5 minutes for the user to complete the MFA challenge. If the DSID cookie is not obtained within that window, the operation fails and an error is returned.

    When MFA is not needed

    If a profile has MFAEnabled: false, the username/password from the OS keychain is used directly in the EAP authentication exchange; no browser is opened.

    If MFAEnabled: true but a valid DSID cookie is already cached in the session parameters (from a previous vpnconnect mfa run), the browser is not opened again.

    Limitations

    • In sso and fido2 modes, only Microsoft Azure AD / Entra SSO flows are supported. Other SAML or OIDC identity providers are not tested.
    • The CSS/XPath selectors used in sso and fido2 modes are hard-coded and may break if Microsoft changes the sign-in page layout. Manual mode has no such limitation.
    • The browser runs on the desktop; headless mode is not used because MFA challenges require user interaction.
