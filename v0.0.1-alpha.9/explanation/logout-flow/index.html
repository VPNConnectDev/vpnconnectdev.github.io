    Logout and session termination

    This document explains how VPNConnect terminates a VPN session both on the client side and on the server side, and why using a proper logout sequence matters.

    Why a proper logout matters

    Simply closing the TLS connection (or killing the process) leaves the session alive on the Pulse Secure server. This causes two problems:

    1. Resource waste: the server holds onto the session slot, firewall rules, and allocated IP address until its own idle-timeout expires.
    2. Session-limit errors: many Pulse Secure deployments impose a limit on concurrent sessions per user. An orphaned session counts against this limit. On the next connection attempt the user may see a "maximum sessions reached" dialog and need to manually evict the old session (this is also what the MFA browser's session-limit handling covers — see MFA browser automation).

    Sending an explicit logout packet mirrors the behaviour of the official Pulse Secure client and of OpenConnect, on whose implementation vpnconnect's logout is modelled.

    Dual-method logout

    vpnconnect attempts logout using two complementary methods in sequence:

    Disconnect() called
       │
       ▼
logoutIFT()  —  send IF-T/TLS packet (vendor=Juniper, type=0x89)
       │
       │  if the TLS connection is already closed or the packet fails
       ▼
logoutHTTP()  —  GET https://<server>/dana-na/auth/logout.cgi
       │
       ▼
Close TLS connection, stop data channel, remove routes, destroy TUN interface

    Errors from either logout step are logged but do not prevent the rest of the disconnect sequence from completing.

    IF-T/TLS logout packet

    The primary method sends a Juniper-specific IF-T/TLS control packet:

    Vendor:   0x0a4c  (Juniper Networks)
Type:     0x89    (Logout / IFTClientLogout)
Length:   16      (header only, no payload)
Sequence: current IF-T sequence number
Payload:  (empty)

    This constant (IFTClientLogout = 0x89) is defined in internal/pulse/ift/packet.go and matches the value used by OpenConnect.

    HTTP logout fallback

    If the IF-T/TLS packet cannot be sent (e.g. the TLS connection was already closed by the server), vpnconnect falls back to an HTTP GET request:

    GET /dana-na/auth/logout.cgi HTTP/1.1
Host: <vpn-server>
Cookie: DSID=<session-cookie>

    This endpoint is standard across Pulse Secure / Ivanti Connect Secure installations and also mirrors OpenConnect's oncp_bye() function. It works even after the IF-T tunnel has been torn down, as it is a plain HTTPS request.

    Sequence number tracking

    To construct valid IF-T packets after authentication, vpnconnect tracks a uint32 sequence counter (iftSeq) inside the Client struct. This counter is incremented for each IF-T packet sent, including the logout packet, ensuring the server accepts it as a legitimate continuation of the session.
